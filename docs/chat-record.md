**定稿基线**

- 架构采用 `模块化单体 API + 独立 worker`。对外接口是 `REST + SSE`，后台慢任务交给 worker。
- 产品是“用户私有知识库问答系统”，不是管理员统一维护的公共知识库平台。

- 会话与知识库是 `1:0..1` 关系，一个会话最多绑定一个知识库。
- 问答链路是标准 `RAG`：先检索，再生成；检索不到时允许回退到通用模型回答，但前端必须明确提示“未命中知识库”。
- 重新生成采用“覆盖当前回答”，不保留旧回答正文；但最新回答仍记录 `model_used`、token 用量、命中的引用块。
- 知识库只做正式文档上传，不做聊天临时文件。
- 文档格式支持 `PDF / TXT / Markdown / DOCX`。
- 分块策略采用“先按标题/段落切，再对超长段落做窗口切分”。
- embedding 模型是“系统给默认值，知识库创建时固化”；之后如果切模型，必须重建索引。
- 存储采用 `PostgreSQL + pgvector + S3 兼容对象存储`。
- 任务系统统一走 `tasks` 表 + worker，支持自动重试 3 次，失败后管理员手工重试。
- 管理后台包含配额、审计、失败任务重试；管理员只能看用户知识库的元数据和任务状态，不能看正文。

**这会怎样改写原稿**

- `sessions.knowledge_base_ids UUID[]` 会改成单个 `knowledge_base_id`。
- `/auth/logout` 不再只是“前端删 token”，而是服务端废弃当前 `refresh token`。
- 需要新增 `user_sessions` 或 `refresh_tokens` 表，用于登录会话和注销失效。
- 需要新增 `tasks` 表，统一表示文档解析、索引重建、清理任务、失败重试。
- 需要新增 `message_citations` 表，按“文档块”记录引用；前端首期只显示文档名。
- 审计日志和运行日志要分开：数据库里重点存审计和业务事件，不直接拿它代替程序运行日志。
- 配额会落到“每日 token、知识库存储容量、文档数量”三类，并支持接近上限告警、超限拒绝。

**你从 Spring Boot 转到 Go 时，最需要理解的差别**

- Go 里也有 `handler / service / repository` 分层，只是依赖通常在 `main` 里显式组装，不靠大量注解自动注入。
- `worker` 可以理解成“脱离 HTTP 请求单独跑的后台任务进程”，很像在 Java 里单独拉出来的异步消费或定时任务服务。
- 大模型接入建议单独做 `provider` 抽象层。业务层只关心“生成回答、做 embedding、统计 token”，不直接绑死 DeepSeek SDK。
- RAG 项目里，聊天和知识库其实是两条链路：`聊天链路` 管会话和流式输出，`知识库链路` 管上传、解析、分块、向量化、检索。
