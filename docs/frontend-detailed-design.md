## 3. 前端总体设计结论

### 3.1 应用形态

前端采用“单仓库双应用”方案：

- 用户端应用：面向普通用户，负责登录、会话、知识库、个人中心、关于页
- 管理端应用：面向管理员，负责 Dashboard、用户管理、模型配置、任务、审计和系统设置

这样设计的原因：

- 用户端和管理端角色边界明显，页面风格和信息密度差异较大
- 当前原型只覆盖用户端，先天不适合把两类界面揉成一个单一 SPA
- 后续打包、权限控制、菜单结构和部署策略更清晰

### 3.2 技术栈

前端基线固定如下：

- 框架：`Vue 3 + TypeScript + Vite`
- 包管理：`pnpm`
- 状态管理：`Pinia`
- 服务端数据管理：`Vue Query`
- 组件库：`Element Plus`
- 样式：`TailwindCSS + SCSS`
- 测试：`ESLint + Prettier + Vitest + Playwright`

### 3.3 联调与鉴权基线

- 当前联调以“后端真实实现”为准
- 开发和部署形态优先采用：
  - 同域部署
  - 或前端开发服务器通过 `dev proxy` 代理到后端
- `refresh token` 继续使用后端返回的 `HttpOnly Cookie`
- `access token` 存放在 `localStorage`
- 应用启动时优先调用 `/auth/refresh` 获取新 `access token`

### 3.4 当前阶段差异处理

由于聊天和管理端后端能力尚未完全就绪，前端设计区分为两层：

- 目标产品结构：最终 V1 应该具备的页面、模块和交互
- 当前开发阶段策略：在后端尚未就绪时如何以 mock 或占位方式推进前端

### 4.2 分层原则

调用方向固定为：

```text
page / layout -> module composables -> shared api client -> generated sdk
```

约束如下：

- 业务页面不直接写裸请求
- `Pinia` 只保存真正跨页面共享的客户端状态
- 列表、详情、分页、轮询等服务端状态优先使用 `Vue Query`
- `Element Plus` 主要用于表单、弹层、表格、通知、上传等通用控件
- 聊天区、主布局、卡片、视觉容器优先自定义

### 4.3 共享层职责

`packages/shared` 只承担基础层职责，不承担完整业务模块职责。

共享层包含：

- API SDK 与请求客户端
- 认证工具和 token 管理
- 通用类型
- 通用组件
- 公共 hooks/composables
- 样式变量、主题 token 和混入

不建议放入共享层的内容：

- 用户端特有业务页面
- 管理端特有业务页面
- 领域强耦合 UI 组件，如“知识库卡片列表页整体容器”

## 5. 用户端信息架构

### 5.1 用户端路由结构

推荐路由如下：

| 路由 | 页面 | 说明 |
|---|---|---|
| `/login` | 登录页 | 未登录入口 |
| `/register` | 注册页 | 未登录入口 |
| `/` | 聊天工作区入口 | 默认进入最近会话或空会话态 |
| `/sessions/:sessionId` | 会话详情页 | 主聊天工作区 |
| `/knowledge-bases` | 知识库列表页 | 知识库入口 |
| `/knowledge-bases/:kbId` | 知识库详情页 | 文档管理与设置入口 |
| `/about` | 关于页 | 登录后可访问 |
| `/me/profile` | 个人资料页 | 个人中心 |
| `/me/security` | 安全设置页 | 修改密码 |

### 5.2 用户端全局布局

用户端核心布局采用“两层结构”：

- 顶层布局负责：
  - 路由鉴权
  - 页面背景和主题容器
  - 全局通知
  - 全局 loading 和错误边界
- 聊天工作区布局负责：
  - 左侧会话/导航栏
  - 主聊天区
  - 知识库绑定抽屉

### 5.3 聊天工作区布局

聊天工作区左侧栏固定为三段结构：

1. 顶部：会话列表
2. 中部：一级导航
3. 底部：用户头像/昵称入口，点击进入个人中心

中部一级导航当前固定为：

- `知识库`
- `关于`

说明：

- “聊天”本身不作为一级导航项，而是由顶部会话列表承担
- 左侧导航更像聊天工作区之外的功能入口

### 5.4 聊天页目标结构

聊天页主区由以下区块组成：

1. 顶部状态条
   - 当前会话名
   - 当前绑定知识库名称
   - 切换知识库入口
2. 消息流区域
   - 助手欢迎消息
   - 历史用户/助手消息
   - 助手消息下方显示引用文档名
3. 输入区
   - 多行输入框
   - 发送按钮
   - 未来预留停止生成入口

消息表现规则：

- 用户消息右对齐，不显示头像
- 助手消息左对齐，显示头像
- “未命中知识库”时，在助手消息头部以标签形式提示
- 如果会话没有绑定知识库，前端不额外增加额外提示文案

### 5.5 聊天页当前开发阶段策略

由于后端聊天主链路尚未实现，当前开发阶段采用以下策略：

- 用本地假数据展示消息区结构、气泡、引用区和欢迎语
- 发送输入区保持可见，但按钮禁用
- 页面中使用产品化占位提示，而不是开发者式大段说明

当后端接口完成后，替换策略如下：

- 用真实会话详情接口替换本地消息假数据
- 接入真实 `SSE`
- 补齐停止生成、重生成和流式错误提示

### 5.6 知识库抽屉职责

聊天页中的知识库抽屉只承担一项职责：

- 选择当前会话绑定的知识库

明确不承担的职责：

- 知识库 CRUD
- 文档管理
- 文档状态查看
- 批量知识库配置

这些能力统一放在知识库页面中完成。

### 5.7 知识库列表页

知识库列表页采用独立页面，不做抽屉式工作台。

页面区块建议：

1. 顶部标题区
   - 页面标题
   - 简短说明
   - “新建知识库”按钮
2. 搜索与筛选区
   - 关键字搜索
3. 列表区
   - 知识库卡片或列表
   - 每项显示名称、描述摘要、文档数量、最近更新时间
4. 空状态区
   - 初次使用引导

### 5.8 知识库详情页

知识库详情页结构固定为：

1. 顶部知识库信息区
   - 名称
   - 描述
   - embedding model
   - 最后索引时间
   - 编辑、删除、重建索引按钮
2. 文档上传区
   - 拖拽上传
   - 点击上传
   - 上传说明
3. 文档列表区
   - 文档名
   - 状态
   - 分块数
   - 更新时间
   - 操作

文档状态处理策略：

- 对 `pending / processing / deleting` 状态启用轮询
- 推荐轮询间隔：`5s`
- `pdf` 首期不在上传前拦截，按后端返回结果提示失败

### 5.9 注册页

注册页采用独立页面，与登录页共享同一套视觉语言。

字段固定为：

- 用户名
- 邮箱
- 密码
- 确认密码
- 协议勾选

不进入 V1 的内容：

- 邮箱验证码
- 找回密码
- 第三方登录

### 5.10 个人中心

个人中心拆为两个页面：

- `/me/profile`
- `/me/security`

#### 5.10.1 个人资料页

建议区块：

- 基础信息展示
  - 用户名
  - 邮箱
  - 角色
- 可编辑信息
  - 昵称

当前不进入 V1 的内容：

- 头像上传
- 头像 URL 编辑
- 个人偏好系统

#### 5.10.2 安全设置页

只承载修改密码能力：

- 原密码
- 新密码
- 确认新密码

前端必须处理的后续行为：

- 修改密码成功后清理本地 `access token`
- 跳回登录页
- 提示用户重新登录

### 5.11 关于页

关于页为登录后页面，承担三类内容：

- 产品介绍
- 使用帮助
- 项目/版本信息

推荐页面结构：

1. 产品介绍区
2. 快速上手区
3. 常见问题区
4. 版本与构建信息区

## 7. 状态管理与数据流设计

### 7.1 `Pinia` 与 `Vue Query` 的边界

推荐边界如下：

| 类型 | 使用方案 |
|---|---|
| 登录态、当前 token、布局开关、主题偏好 | `Pinia` |
| 列表、详情、分页、轮询、创建/更新/删除后的数据同步 | `Vue Query` |
| 表单临时值、弹层开关、局部 UI 状态 | 组件本地状态 |

### 7.2 推荐 Store 划分

- `authStore`
  - `accessToken`
  - 当前用户摘要
  - 登录态恢复状态
- `appStore`
  - 布局折叠状态
  - 全局抽屉/弹层状态
- `chatUiStore`
  - 当前侧栏选中状态
  - 当前知识库抽屉开关

### 7.3 Query Key 约定

推荐统一 Query Key：

- `['me']`
- `['sessions', filters]`
- `['session', sessionId]`
- `['knowledge-bases', filters]`
- `['knowledge-base', kbId]`
- `['documents', kbId, filters]`
- `['admin', 'users', filters]`
- `['admin', 'tasks', filters]`

## 8. API 与鉴权设计

### 8.1 接口真相源策略

当前阶段采用以下规则：

1. 以当前后端真实实现为联调真相源
2. 前端可维护一份与当前实现一致的接口描述快照用于生成 SDK
3. 当后端正式补齐 `OpenAPI` 后，再回收为统一契约来源

这可以同时满足两件事：

- 当前联调不被过时草案误导
- 仍保留“代码生成客户端”的工程收益

### 8.2 API SDK 结构

推荐分三层：

1. `generated sdk`
   - 由接口描述生成
2. `api client wrapper`
   - 统一 `baseURL`
   - 统一 headers
   - 统一错误转换
   - 自动携带 token
   - 自动处理 `credentials`
3. `module service adapter`
   - 面向页面和模块的业务语义方法

### 8.3 鉴权流程

#### 8.3.1 应用启动

1. 应用启动
2. 调用 `/auth/refresh`
3. 若成功：
   - 保存新 `access_token`
   - 拉取 `/users/me`
   - 进入业务页面
4. 若失败：
   - 清空本地 `access token`
   - 跳转登录页

#### 8.3.2 登录

1. 提交账号密码
2. 后端返回：
   - `access_token`
   - `user`
   - `HttpOnly refresh cookie`
3. 前端保存 `access_token`
4. 跳转到聊天工作区

#### 8.3.3 请求鉴权

- 所有登录态接口统一在请求层注入 `Authorization: Bearer <token>`
- 请求客户端统一带上 `credentials`

#### 8.3.4 401 处理

推荐策略：

1. 首次遇到 `401`
2. 尝试一次 `/auth/refresh`
3. 若刷新成功，重放原请求
4. 若刷新失败，清理本地状态并跳转登录页

#### 8.3.5 退出登录

- 调用 `/auth/logout`
- 清空本地 `access token`
- 清空用户态
- 跳转登录页

### 8.4 错误处理规范

后端错误包裹结构包含：

- `code`
- `message`
- `request_id`
- `details`

前端处理规范：

- 字段级校验错误进入表单项提示
- 全局业务错误使用 `Element Plus` 的 message/notification
- 需要排查的问题在开发环境打印 `request_id`

### 8.5 文件上传规范

- 上传使用原生 `multipart/form-data`
- 上传后立即进入文档列表轮询
- 大文件或失败状态需要在列表中有清晰状态反馈

### 8.6 SSE 设计预留

当前阶段先不实现真实 SSE，但前端架构必须预留：

- SSE 连接建立与关闭
- `meta`、`delta`、`done`、`error` 事件解析
- 流式文本拼接
- 中断状态恢复

建议把 SSE 解析放在独立模块中，不散落到页面组件里。

## 9. 视觉系统设计

### 9.1 总体风格

用户端视觉方向固定为：

- 轻柔
- 空气感
- 插画气质
- 高级化处理
- 沉浸式 AI 对话空间

管理端视觉方向固定为：

- 与用户端保持同品牌感
- 颜色和材质更克制
- 更强调信息密度、稳定感和操作清晰度

### 9.2 色彩建议

推荐建立以下主题 token：

| Token | 建议用途 |
|---|---|
| `--color-brand-sky` | 品牌主色，偏雾蓝 |
| `--color-brand-night` | 深色文本和重点分隔 |
| `--color-surface-frost` | 玻璃面板底色 |
| `--color-surface-soft` | 普通卡片底色 |
| `--color-text-primary` | 主文本 |
| `--color-text-secondary` | 次文本 |
| `--color-success` | 可用/成功状态 |
| `--color-warning` | 处理中/警告状态 |
| `--color-danger` | 失败/删除状态 |

视觉落地原则：

- 用户端避免纯白硬底
- 多用半透明浅色面板、柔和边框和模糊背景
- 管理端收敛玻璃效果，更多使用稳定的面板和网格

### 9.3 字体建议

字体方向固定为“有识别度的 Web 字体组合”。

推荐组合：

- 标题：`Noto Serif SC` 或同类高识别度中文衬线字体
- 正文/UI：`Noto Sans SC` + 中文系统字体回退
- 代码与数字：`JetBrains Mono`

原则：

- 标题与正文要有明确层级差
- 中文阅读必须优先保证稳定性
- 具体字体资源在实施时再根据可用性和加载成本确认

### 9.4 登录页视觉规范

登录页使用 [登录界面背景图.jpg](/home/kitra/Projects/SoftwareEngineeringCourseDesign/frontend/UI原型/登录界面背景图.jpg) 作为基础背景。

必须加入以下视觉处理：

- 覆盖整屏黑色半透明玻璃层，降低背景亮度
- 中央悬浮玻璃卡片承载表单
- 卡片应具备：
  - 轻微模糊
  - 低对比高质感边框
  - 较大留白

建议不直接复刻原型中的简陋输入框和按钮比例，而是做以下优化：

- 表单控件更统一
- 标题与副标题层级更清晰
- 错误提示内聚在卡片内

### 9.5 聊天页视觉规范

聊天页强调沉浸式、安静、连续的阅读体验。

设计原则：

- 以消息内容为主，而不是重装饰
- 助手消息显示头像，用户消息不显示头像
- 输入区固定但不过度抢眼
- 会话列表和导航区要弱于主消息区

建议补强点：

- 顶部知识库状态条采用低对比信息条
- 助手消息容器和引用列表有轻微分层
- 消息出现可做轻微渐入

### 9.6 无原型页面的低保真原则

没有原型的页面不追求先做高保真，而是先满足：

- 信息层级明确
- 操作路径短
- 与用户端整体品牌一致

## 10. 代码规范与工程约定

### 10.1 命名规范

- 组件文件：`PascalCase.vue`
- hooks/composables：`useXxx.ts`
- store：`useXxxStore.ts`
- module 目录使用业务域命名，不使用 `misc`、`common2` 等模糊命名

### 10.2 组件拆分原则

满足以下任一条件时应拆组件：

- 同一区块在两个页面复用
- 单组件超过一个清晰职责
- 模板结构过长，难以扫描
- 包含复杂交互或状态切换

### 10.3 样式规范

- 布局、间距、栅格优先使用 `TailwindCSS`
- 复杂状态、动画、主题变量和少量长样式使用 `SCSS`
- 不建议在模板里堆叠过长 class 串；必要时抽成语义类

### 10.4 `Element Plus` 使用边界

推荐使用：

- 表单
- 表格
- Dialog
- Drawer
- Upload
- Message / Notification

不建议直接套默认外观的区域：

- 整个聊天消息区
- 整个登录页主视觉
- 主工作区外框

### 10.5 测试规范

- `ESLint`：基础静态检查
- `Prettier`：格式统一
- `Vitest`：工具函数、composables、关键状态逻辑
- `Playwright`：登录、知识库 CRUD、上传、路由守卫等主流程

## 13. 结论

V1 前端采用“单仓库双应用 + 共享基础层”的架构最稳妥：

- 用户端负责会话、知识库、个人中心和关于页
- 管理端负责 Dashboard 和后台管理能力
- 当前实施先以用户端基础骨架和知识库模块为先
- 聊天主链路和管理端真实联调随着后端能力到位再逐步接入

在此基础上，前端已经具备进入工程初始化和后续编码阶段的设计前提。
